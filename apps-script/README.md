# Apps Script 백엔드 배포 가이드

이 문서는 처음 해보는 분도 따라할 수 있도록 단계별로 자세하게 작성했습니다.
순서대로 따라가시면 됩니다. 막히면 각 단계 아래의 "막혔을 때" 항목을 확인해주세요.

> ⚠️ **주의**: 이 백엔드는 **서울고 교실 사진 백엔드와 별개의 새 프로젝트**입니다.
> 기존 사진 앱 백엔드를 수정하지 말고, 아래 1단계처럼 새 프로젝트를 만드세요.

---

## 0. 사전 준비물

- Google 계정 (`igodswill.org` 워크스페이스 계정 권장)
- 별도의 Google Drive 폴더는 필요 없습니다 (이 백엔드는 PIN 인증만 담당, 파일 입출력 없음).

---

## 1. Apps Script 새 프로젝트 만들기

1. 브라우저에서 [https://script.google.com](https://script.google.com) 접속
2. 좌측 상단 **"+ 새 프로젝트"** 클릭
3. 상단의 **"제목 없는 프로젝트"** 부분을 클릭하여 이름을 `교회 인트라넷 백엔드`로 변경

> 💡 서울고 교실 사진 백엔드(`교회 교실 사진 백엔드`)와 **이름이 다릅니다**. 두 프로젝트를 헷갈리지 마세요.

### 1-1. 코드 붙여넣기

1. 좌측 파일 트리에서 기본으로 생긴 `Code.gs` 클릭
2. 편집창의 모든 내용을 지우고, 이 저장소의 [`apps-script/Code.gs`](./Code.gs) 파일 전체를 복사하여 붙여넣기
3. 좌측 상단의 💾(저장) 아이콘 또는 `Cmd+S` / `Ctrl+S`로 저장

### 1-2. 매니페스트 파일 동기화 (권장)

시간대를 KST로 정확히 맞추기 위해 권장합니다.

1. 좌측 톱니바퀴 모양의 **"프로젝트 설정"** 클릭
2. **"'appsscript.json' 매니페스트 파일을 편집기에 표시"** 체크박스 켜기
3. 좌측 파일 트리로 돌아오면 `appsscript.json`이 보임
4. 그 파일 내용을 [`apps-script/appsscript.json`](./appsscript.json) 내용으로 교체하고 저장

## 2. 스크립트 속성 설정 (PIN)

이 부분이 빠지면 **누구나 임의의 PIN으로 통과**(개발 모드)되므로, 운영 시작 전에 꼭 설정하세요.

1. 좌측의 톱니바퀴 모양 **"프로젝트 설정"** 클릭
2. 페이지 아래쪽 **"스크립트 속성"** 섹션의 **"스크립트 속성 추가"** 버튼 클릭
3. 다음과 같이 입력:
   - **속성**: `PIN`
   - **값**: `9999`  ← 임시 PIN. 운영 안정화 후 변경 권장.
4. **"스크립트 속성 저장"** 클릭

> 💡 **`SECRET` 속성**은 첫 토큰 발급 시 자동 생성됩니다. 직접 만들 필요 없습니다.
> - 이 키가 노출되면 토큰 위조가 가능해지므로, 외부에 공유하지 마세요.
> - 의심스러우면 `SECRET` 값을 삭제하면 자동 재생성되며 모든 기존 토큰이 무효화됩니다.

## 3. 동작 테스트 (권한 부여)

이 백엔드는 Drive 권한이 필요 없으므로 권한 승인 단계가 가볍습니다.

1. 편집기로 돌아와 좌측의 함수 선택 드롭다운에서 **`testSetup`** 선택
2. **"실행"** 버튼 클릭
3. 처음에는 권한 승인 창이 뜸:
   - **"권한 검토"** → 사용할 Google 계정 선택
   - "이 앱은 Google에서 인증되지 않았습니다" 경고가 뜨면:
     - **"고급"** 클릭 → **"교회 인트라넷 백엔드(안전하지 않음)으로 이동"** 클릭
     - (본인이 만든 코드이므로 안전합니다)
   - 요청 권한들 확인 후 **"허용"**
4. 다시 한 번 **"실행"** 클릭 → 하단 **실행 로그** 탭에 다음과 비슷한 출력이 뜨면 성공:
   ```
   PIN 설정 여부: 예
   SECRET 자동 생성됨: true
   서버 시간(KST): 2026-04-29T...
   ```

5. (선택) 토큰 검증 동작 확인: 함수 드롭다운에서 **`testTokenRoundTrip`** 선택 후 실행. 다음 비슷한 로그가 보이면 OK:
   ```
   토큰: eyJleHAiOj...AbCdEf...
   verify(현재 PIN 기준): true
   verify(다른 PIN 기준, false 기대): false
   ```

## 4. 웹앱으로 배포

이제 외부(브라우저)에서 호출 가능한 URL을 만듭니다.

1. 편집기 우측 상단의 **"배포"** 드롭다운 → **"새 배포"** 클릭
2. 배포 다이얼로그 좌측의 톱니바퀴 아이콘 → **"웹 앱"** 선택
3. 다음과 같이 설정:
   - **설명**: `v1` (선택)
   - **다음 사용자 인증 정보로 실행**: `나` (배포한 본인 계정)
   - **액세스 권한이 있는 사용자**: **`모든 사용자`** ← 중요
4. **"배포"** 클릭
5. 배포가 완료되면 **"웹 앱 URL"**이 표시됨. 형식은:
   ```
   https://script.google.com/macros/s/AKfycby......./exec
   ```
6. 이 URL을 복사하여 어딘가에 임시 저장 (5번 단계에서 프론트에 붙여넣음)

### 코드를 수정해서 다시 배포하고 싶을 때

- **"배포"** → **"배포 관리"** → 기존 배포 옆 ✏️(연필) 아이콘 → **"새 버전"** 선택 → **"배포"**
- 이렇게 하면 **URL이 바뀌지 않습니다**. (새 배포로 만들면 URL이 새로 생성됨)

## 5. 프론트엔드에 API URL 연결

방법 두 가지 중 편한 쪽으로 선택하세요.

### 방법 A: 코드에 직접 박기 (배포 후 권장)

`js/api.js` 파일을 열고 다음 줄을 찾아 URL을 채워 넣습니다.

```javascript
const API_URL = '';  // ← 여기
```

```javascript
const API_URL = 'https://script.google.com/macros/s/AKfy.../exec';
```

저장하고 GitHub에 push (GitHub Pages 배포 시).

### 방법 B: 브라우저에서 임시로 덮어쓰기 (개발/테스트용)

브라우저에서 앱 페이지를 연 다음 개발자 도구(콘솔)에서 한 번만 실행:

```javascript
localStorage.setItem('API_URL_OVERRIDE', 'https://script.google.com/macros/s/AKfy.../exec')
```

이후 새로고침하면 그 URL을 사용합니다.

## 6. 동작 확인

1. 로컬 서버 띄우기 (포트는 8081 — 서울고 앱이 8080을 쓰는 중일 수 있음):
   ```bash
   cd staff-portal
   python3 -m http.server 8081 --bind 0.0.0.0
   ```
2. 브라우저에서 `http://localhost:8081` (또는 같은 와이파이의 모바일에서 `http://<맥IP>:8081`)
3. PIN 화면이 뜸 → `9999` 입력 → 메뉴 카드 4개가 보이는 홈 화면으로 이동되면 성공

### 막혔을 때

- **"API_URL이 설정되지 않았습니다"** → 5번 단계에서 URL 등록 안 됨
- **CORS 오류** → 4번에서 "액세스 권한"이 "모든 사용자"가 아님. 재배포 필요.
- **"PIN이 틀렸습니다"** → 2번 단계에서 `PIN` 속성이 `9999`로 정확히 들어갔는지 확인. 앞뒤 공백 주의.
- **PIN 화면이 안 뜨고 바로 홈으로** → 백엔드 PIN 미설정(개발 모드) 상태. 2번 단계에서 PIN 속성을 추가하면 됨.

## 7. PIN 변경 / 운영

- **PIN 변경**:
  1. Apps Script 편집기 → 좌측 **프로젝트 설정**(톱니바퀴) → 스크립트 속성에서 `PIN` 값 수정 → 저장
  2. 재배포 **불필요**. 변경 즉시 반영됨.
  3. **자동 효과**: 모든 기존 사용자의 토큰이 자동 무효화 → 다음 접속 시 새 PIN으로 다시 로그인 필요.
  4. 새 PIN을 교직원분들께 안내해주세요.
- **PIN을 잊었을 때**: 위와 동일하게 `PIN` 속성 값을 새로 정해서 덮어쓰면 됩니다.
- **모든 토큰을 한꺼번에 강제 만료시키고 싶을 때**: 스크립트 속성에서 `SECRET` 값을 삭제 → 다음 토큰 발급 시 자동 재생성되고 기존 토큰은 모두 검증 실패.
- **메뉴 추가/수정/삭제**: `js/menus.json` 편집 후 GitHub에 push (백엔드 변경 없음).
- **서버 코드 변경**: 4번의 "코드를 수정해서 다시 배포..." 참고.

## 부록 A: API 엔드포인트 명세

### POST `/exec` (action: auth)

**요청 (Content-Type: text/plain;charset=utf-8 — CORS 프리플라이트 회피용)**
```json
{ "action": "auth", "pin": "9999" }
```

**응답 (성공)**
```json
{
  "success": true,
  "token": "eyJleHAiOjE3..ZX0.AbCdEf...",
  "expiresAt": 1759999999999
}
```

**응답 (실패)**
```json
{ "success": false, "error": "PIN이 틀렸습니다" }
```

**토큰 형식**: `base64url(payload).base64url(HMAC-SHA256 sig)`
- `payload = { exp, iat, v }` — `v`는 PIN의 짧은 SHA-256 해시
- 서명 키는 자동 생성되는 `SECRET` 속성
- PIN이 바뀌면 `v` 불일치로 자동 무효화

### GET `/exec?action=ping`

서버 동작 확인용 + PIN 설정 여부.

```json
{
  "success": true,
  "ping": "pong",
  "time": "2026-04-29T...",
  "pinRequired": true
}
```

`pinRequired: false`이면 프론트는 PIN 화면을 스킵하고 dev 토큰을 자동 발급합니다.

## 부록 B: 향후 확장 메모

지금 백엔드는 PIN 인증만 담당합니다. 향후 staff-portal에 무거운 기능(예: 관리자 페이지에서 메뉴 편집)이 필요하면 다음 패턴으로 확장하세요.

- `doPost`/`doGet`의 `action` 분기에 핸들러를 추가하고
- 인증이 필요한 액션은 진입점에서 `verifyToken_(body.token)` (또는 `params.token`)으로 검증

`Code.gs`에 `verifyToken_` 함수가 이미 준비되어 있어 바로 사용 가능합니다.
